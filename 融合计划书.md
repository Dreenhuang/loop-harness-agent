# Boss-auto-harness 与 Loop Agent 融合计划书

> **文档版本**：v1.0  
> **编制日期**：2026-06-16  
> **适用范围**：`g:\ai-gongju\Loop-agent` 及后续基于 Loop Agent 的自动化研发项目  
> **编制目标**：为 `boss-auto-harness` 与 `Loop Agent` 的体系融合提供可执行、可验证、可落地的技术指导

---

## 1. 背景与目标

`Loop Agent` 已具备 4 级封装架构、16 角色协作、10 相位工作流、4 道质量门禁、黑板+A2A 协议、无人值守模式与 Domain Chip 自进化能力，适合承担复杂项目的全流程自动化调度。`boss-auto-harness` 则强调 Harness Engineering 的规范开发方法，把 `brainstorming`、`writing-plans`、`test-driven-development`、`verification-before-completion`、`systematic-debugging` 等纪律固化为 11 步 Pipeline，强制形成“需求澄清 → 设计先行 → 微任务拆解 → TDD 实施 → 验证闭环 → 审查闭环 → 发布闭环”的规范工程链路。

两者并非替代关系，而是“调度编排体系”与“执行纪律体系”的互补关系：

- `Loop Agent` 强在角色分工、状态管理、门禁治理、流程推进、跨 Agent 协调。
- `boss-auto-harness` 强在单任务执行纪律、文档工件完整性、TDD 约束、完成前验证、设计优先和可追踪研发方法。

本计划书目标是将 `boss-auto-harness` 作为 `Loop Agent` 的“执行方法层”和“过程质量层”进行内嵌式融合，使 Loop Agent 在不破坏现有架构的前提下，获得更强的执行规范性、研发可审计性和交付质量稳定性。

## 2. 现状分析

### 2.1 Loop Agent 现状能力概述

当前 `Loop Agent` 的核心结构可概括为：

- **架构层**：Skill → Agent Profile → Workflow Blueprint → Domain Chip 四层封装。
- **调度层**：`@Orchestrator` 负责 phase 推进、任务分派、预算管理、快照和异常调度。
- **执行层**：16 个角色按职责拆分需求、设计、架构、开发、测试、知识沉淀、文档和部署工作。
- **控制层**：3 道硬刹车负责预算、迭代和无进展检测，4 道门禁负责代码、性能、测试和最终发布质量控制。
- **状态层**：通过黑板 `项目进度记录.md` 统一记录阶段、节点、依赖、门禁、快照和风险。
- **协议层**：A2A 协议保证 Agent 之间通过标准消息格式进行协作。
- **演进层**：Domain Chip 中的 benchmark 与 autoloop 支持基于评估集的自改进。

Loop Agent 的优势是“能调度、能推进、能治理、能约束流程边界”；不足在于“对单个执行任务内部如何高质量完成”的方法学约束还不够细，尤其在需求澄清深度、微任务粒度、TDD 强制性、完成声明证据化、审查反馈闭环等方面仍然依赖执行者自觉。

### 2.2 boss-auto-harness 现状能力概述

`boss-auto-harness` 的核心价值在于把开发活动结构化为 11 个不可跳过的步骤，并为每一步绑定明确的技能纪律与输出工件：

- Step 1：`brainstorming` + `product-spec-builder`
- Step 2：设计规范收敛
- Step 3：UI 设计先行
- Step 4：组件库与设计实现
- Step 5：`writing-plans` 微任务分解
- Step 6：`test-driven-development` + `subagent-driven-development` + `using-git-worktrees`
- Step 7：`verification-before-completion` + `quality-gate`
- Step 8：测试执行与覆盖率治理
- Step 9：代码审查请求与反馈处理闭环
- Step 10：UX 审查
- Step 11：发布构建与分支完成验证

它的优势是“方法学扎实、文档驱动明确、执行纪律强”；不足在于原始设计更偏向单项目开发框架，对多角色协同、黑板状态同步、跨 Phase 全局治理、门禁级回退和无人值守资源编排的原生支持不如 Loop Agent 完整。

### 2.3 融合定位

推荐的融合定位不是“双主脑并存”，而是采用“**Loop Agent 做外层总控，boss-auto-harness 做内层执行规范引擎**”的主从式架构：

- `Loop Agent` 保留 `@Orchestrator`、黑板、A2A、4 层封装、10 Phase 和 4 Gate 的主导权。
- `boss-auto-harness` 以“执行协议”“阶段模板”“质量补强策略”“工件标准”的形式嵌入各个角色与 phase 内部。
- 对用户暴露的入口仍然以 `Loop Agent` 为主，避免触发入口分裂。

## 3. 技术可行性分析

### 3.1 融合可行性结论

总体判断：**技术上高度可行，推荐推进，风险可控。**

原因如下：

1. 两套体系的核心理念一致，均强调流程不可跳过、质量门禁、阶段性工件、可追踪交付与角色边界。
2. `Loop Agent` 的 Phase 结构与 `boss-auto-harness` 的 11 步 Pipeline 在语义上高度对应，可建立稳定映射关系。
3. `boss-auto-harness` 的能力大多以 Skill 和过程规则存在，适合作为 `Loop Agent` 的附加治理层接入，无需重写现有调度引擎。
4. `Loop Agent` 本身已具备 Skill 层、Agent Profile 层和 Workflow 层，可天然承载 harness 纪律注入。
5. Domain Chip 的 `input_contract` / `output_contract` 机制为融合后工件扩展和报告标准化提供了扩展点。

### 3.2 兼容性分析

#### 3.2.1 流程兼容性

可建立如下 Phase/Step 对照：

| Loop Agent Phase | boss-auto-harness Step | 融合方式 |
|---|---|---|
| Phase 0 初始化 | Step 0 前置准备 | 在加载资产后增加 Harness 规则注册 |
| Phase 1 需求基线 | Step 1 需求收集 | 强制引入 `brainstorming` 与 `Product-Spec.md` 约束 |
| Phase 2 交互设计 | Step 2 + Step 3 | 在 UX/UI 阶段引入 Design Brief 与 UI Design 工件 |
| Phase 3 视觉设计 | Step 3 + Step 4 | 将组件库和状态规范纳入交付要求 |
| Phase 4 技术架构 | Step 5 前半段 | 架构结果同步为开发计划输入 |
| Phase 5 并行开发 | Step 5 + Step 6 | 引入微任务、TDD、子代理执行和 worktree 隔离 |
| Gate 1 前后 | Step 7 + Step 9 | 把验证前置与代码审查闭环融合进 Gate 1 |
| Gate 2/3 | Step 8 + Step 10 | 用测试与 UX 审查增强现有门禁质量解释力 |
| Phase 7/8/9 | Step 11 + 全程文档 | 在知识沉淀、文档归档、终审中追加 Harness 工件校验 |

结论：流程兼容性高，且映射关系清晰。

#### 3.2.2 角色兼容性

`Loop Agent` 的 16 角色与 `boss-auto-harness` 的执行要求无角色级冲突，但需要职责补充：

- `@Requirements` 增加苏格拉底式澄清责任。
- `@UX-Researcher`、`@UI-Designer` 增加 Design Brief、UI-Design、Component-Library 输出责任。
- `@Backend`、`@Fullstack-Coder` 增加 TDD、微任务、完成前验证责任。
- `@Code-Reviewer` 增加对 Harness 工件完整性与反馈闭环的检查。
- `@Documenter` 负责收口 Harness 新增文档资产。
- `@Knowledge-Curator` 负责沉淀融合执行中的经验模板与反模式。

结论：角色兼容性高，主要是职责扩展，不需要新增大量核心角色。

#### 3.2.3 工件兼容性

Loop Agent 当前输出偏重 PRD、架构、代码、测试、门禁报告和部署结果；boss-auto-harness 要求输出 `Product-Spec.md`、`Design-Brief.md`、`UI-Design.md`、`Component-Library.md`、`DEV-PLAN.md`、`Quality-Check-Report.md`、`Code-Review-Report.md`、`UX-Review-Report.md`、`Release-Notes.md`、`CLAUDE.md`。

这些工件与 Loop Agent 的黑板和 Domain Chip 输出契约并不冲突，只需：

- 扩展 `docs` 输出项的目录结构；
- 在黑板中增加“工件完整性状态”字段；
- 在 Gate 1 / Gate 4 中增加“工件齐全性”检查项。

结论：工件兼容性高，改动集中在文档治理与门禁规则。

### 3.3 潜在冲突点

#### 冲突点 1：流程步数不一致

- `Loop Agent` 以 10 Phase + 4 Gate 为主线。
- `boss-auto-harness` 以 11 Step 为主线。

风险：若简单叠加，会形成双重流程、状态冗余、用户理解成本上升。

解决策略：

- 不增加第二套顶层流程；
- 采用“Phase 内嵌 Harness Step”的映射模式；
- 对外只显示 Loop Phase，对内在 Agent 指令和文档模板中执行 Harness Step。

#### 冲突点 2：入口控制权冲突

- `Loop Agent` 要求所有复杂任务优先经过 `@Orchestrator` 调度。
- `boss-auto-harness` 也可作为独立框架触发。

风险：用户可能绕过 Orchestrator 直接使用 Harness，破坏统一黑板与状态管理。

解决策略：

- 规定在当前仓库中 Harness 只作为 Loop Agent 子协议使用；
- 新增规则：当识别到 `boss-auto-harness` 相关请求时，自动转交 `@Orchestrator` 做 Phase 编排；
- 保留独立 skill 作为执行能力，但不作为顶层流程入口。

#### 冲突点 3：Maker-Checker 分离与 TDD 内部验证的边界冲突

- Loop Agent 强调“生成者不验证自己”。
- Harness 强调 Step 6 中开发者自己先完成 TDD 和完成前验证。

风险：容易被误解为违反职责分离。

解决策略：

- 明确区分“自检”和“正式门禁验证”；
- Step 6/7 中的 TDD、验证前证据是开发自证，不替代 Gate 1/2/3/4；
- 仍由独立 reviewer/tester/final-reviewer 做正式判定。

#### 冲突点 4：文档体量增加导致执行成本上升

风险：新增 8~10 类文档可能拉长周期、增加 token 消耗、降低无人值守效率。

解决策略：

- 把新增文档分为“强制核心工件”和“按项目类型启用工件”；
- 对纯后端项目、纯修复任务使用裁剪模板；
- 文档骨架自动生成，内容由阶段结果自动回填，减少重复编写。

#### 冲突点 5：UI 优先原则与非 UI 项目适配性冲突

风险：Harness 默认 GUI 设计优先，不适用于 CLI、后端服务、基础设施类任务。

解决策略：

- 抽象为“体验/接口契约优先”，而非强制“视觉设计优先”；
- Web/小程序项目保留 UI 设计完整链路；
- 后端/API 项目则将 Step 2~4 替换为 API 协议、数据模型和交互契约设计。

### 3.4 技术可行性结论

建议采用“**轻量接入、渐进增强、以规则和工件扩展为主、以引擎修改为辅**”的融合策略。该方案能最大化复用现有 Loop Agent 基础设施，最小化重构成本，同时保留后续演进空间。

## 4. 融合架构设计

### 4.1 总体设计原则

1. **单总控原则**：`@Orchestrator` 是唯一全局调度入口。
2. **内嵌增强原则**：Harness 作为执行纪律层嵌入现有 Loop Phase。
3. **不破坏现有契约原则**：保留黑板、A2A、Workflow、Domain Chip 的现有模型。
4. **工件标准化原则**：新增工件统一纳入文档资产目录与黑板登记。
5. **门禁前移原则**：尽可能把质量要求前移到开发中和完成前验证阶段。
6. **可裁剪原则**：根据项目类型启用不同级别的 Harness 约束。

### 4.2 融合后的分层架构

建议形成五段式逻辑分层：

1. **入口层**：用户请求、Loop Agent 触发器、无人值守入口。
2. **编排层**：`@Orchestrator` + Workflow Blueprint + Phase State Machine。
3. **执行协议层**：Harness Policy Engine，负责把 Harness Step、Skill、工件要求注入角色执行。
4. **执行层**：16 个 Agent 按增强后的协议执行任务。
5. **状态与评估层**：黑板、A2A、Gate、Benchmark、Knowledge Base。

建议新增概念组件：

- **Harness Policy Engine**：规则映射器，负责 Phase → Step、Role → Discipline、Task → 工件模板的动态注入。
- **Artifact Registry**：统一登记 Harness 新增工件路径、状态、版本、责任人。
- **Execution Evidence Collector**：统一收集 TDD 结果、验证命令、审查反馈、测试覆盖、UX 证据。

### 4.3 模块整合方案

#### 4.3.1 Workflow 层整合

在 `g:\ai-gongju\Loop-agent\.trae\workflows\prd-to-production.json` 的 Phase 元数据中增加以下字段：

- `harness_steps`
- `required_artifacts`
- `required_disciplines`
- `evidence_requirements`

示例逻辑：

- Phase 1：绑定 `brainstorming`、`product-spec-builder`
- Phase 5：绑定 `writing-plans`、`test-driven-development`、`subagent-driven-development`
- Gate 1 前：绑定 `verification-before-completion`

#### 4.3.2 Agent Profile 层整合

在角色配置中增加 Harness 元数据：

- `harness_responsibilities`
- `artifact_outputs`
- `verification_protocol`
- `fallback_mode`

例如：

- `backend.agent.toml` 增加 `must_use_tdd = true`
- `requirements.agent.toml` 增加 `must_use_brainstorming = true`
- `documenter.agent.toml` 增加 `artifact_registry_sync = true`

#### 4.3.3 Blackboard 层整合

在 `项目进度记录.md` 中新增：

- Harness 工件登记表
- Discipline 执行状态表
- 完成前验证证据表
- 审查反馈闭环追踪表

这样可以让 `Loop Agent` 的状态管理能力完整覆盖 Harness 的执行证据。

#### 4.3.4 Domain Chip 层整合

在 `g:\ai-gongju\Loop-agent\domain-chips\web-feature-chip\chip.json` 的输出契约中扩展：

- `artifacts`
- `evidence`
- `compliance_reports`
- `harness_score`

同时把 benchmark 指标中加入：

- 文档完整率
- 验证证据覆盖率
- TDD 执行率
- 审查闭环完成率

### 4.4 接口设计

#### 4.4.1 Orchestrator → Harness Policy Engine

接口目标：在任务派发前，根据当前 phase、任务类型、项目类型生成 Harness 执行协议。

建议接口：

```json
{
  "phase": "Phase 5",
  "project_type": "web-fullstack",
  "task_type": "feature-development",
  "role": "backend",
  "required_harness_steps": ["step5", "step6", "step7"],
  "required_skills": ["writing-plans", "test-driven-development", "verification-before-completion"],
  "required_artifacts": ["DEV-PLAN.md", "Quality-Check-Report.md"],
  "evidence_requirements": ["failing_test", "passing_test", "verification_commands"]
}
```

#### 4.4.2 Agent → Artifact Registry

接口目标：角色完成阶段任务后，将新增或更新的工件登记到统一索引。

建议字段：

- `artifact_name`
- `artifact_path`
- `owner_role`
- `phase`
- `status`
- `version`
- `updated_at`

#### 4.4.3 Agent → Evidence Collector

接口目标：收集 TDD、测试、验证、审查反馈、UX 审查等证据。

建议字段：

- `evidence_type`
- `source_role`
- `task_id`
- `command`
- `result_summary`
- `attachments`
- `timestamp`

#### 4.4.4 Gate → Compliance Checker

接口目标：在 Gate 阶段统一检查“功能质量 + Harness 合规性”。

建议返回：

- `pass/fail`
- `blockers`
- `missing_artifacts`
- `missing_evidence`
- `discipline_violations`

### 4.5 数据流规划

融合后的主数据流建议如下：

1. 用户提交需求。
2. `@Orchestrator` 识别 phase、项目类型、风险等级。
3. Harness Policy Engine 生成该任务的执行协议。
4. 协议随 A2A 任务消息一起派发给目标 Agent。
5. Agent 在执行过程中持续产出工件和证据。
6. 工件写入 Artifact Registry，证据写入 Evidence Collector，并同步黑板。
7. Gate 读取黑板 + Artifact Registry + Evidence Collector 进行合规校验。
8. 若通过，推进 phase；若不通过，回退修复并保留历史证据。
9. Domain Chip 最终输出代码、文档、报告和知识沉淀结果。

### 4.6 模块交互机制

建议采用“编排消息 + 工件回写 + 门禁拉取”的三段式机制：

- **编排消息**：由 `@Orchestrator` 发起，包含 Harness 协议负载。
- **工件回写**：由执行 Agent 将产物与证据写回黑板和注册表。
- **门禁拉取**：由 Reviewer/Tester/Final-Reviewer 从统一状态源拉取检查数据。

这样可以保持 Loop Agent 现有的“统一状态源 + 独立审查者”原则，不形成双重状态系统。

## 5. 功能增强策略

### 5.1 新增能力

融合后可新增以下高价值能力：

#### 5.1.1 Harness 合规模式

系统可在黑板和最终报告中标识当前项目属于：

- 基础 Loop 模式
- Loop + Harness 标准模式
- Loop + Harness 严格模式
- Loop + Harness 无人值守模式

这将为不同复杂度项目提供分层治理能力。

#### 5.1.2 微任务驱动开发

通过把 `writing-plans` 融入开发阶段，可实现：

- 更细粒度的任务拆解
- 更高并行度的子代理执行
- 更准确的进度估算与卡点定位
- 更稳定的无人值守恢复能力

#### 5.1.3 TDD 证据化交付

当前 Loop Agent 有测试和门禁，但缺少对“是否真正按 TDD 开发”的过程化证明。融合后可新增：

- failing test 记录
- green test 记录
- refactor 记录
- TDD 执行率统计

这会显著提升代码正确性和演化安全性。

#### 5.1.4 完成前验证证据链

新增统一证据链后，系统可做到：

- 每个“完成”声明都有命令与结果支持
- 每个 phase 可追溯具体验证动作
- 最终发布报告可直接引用验证证据

#### 5.1.5 审查反馈闭环跟踪

通过引入 `requesting-code-review` 和 `receiving-code-review` 的闭环约束，可实现：

- 审查意见逐条跟踪
- 修复状态透明化
- 复审结果结构化
- Gate 1 不再只给“过/不过”，还能说明“为什么还不能过”

### 5.2 对现有能力的优化方向

#### 5.2.1 对需求阶段的优化

现有 Phase 1 更偏结果导向。融合后通过 `brainstorming` 可增强：

- 需求澄清深度
- 替代方案对比质量
- 需求遗漏率控制
- AI 能力需求识别完整度

#### 5.2.2 对设计阶段的优化

现有 Phase 2/3 可以进一步标准化为：

- Design Brief 先于 UI 出图
- 组件状态与空/错/加载态前置定义
- UI 设计与实现之间形成更强契约

#### 5.2.3 对开发阶段的优化

现有 Phase 5 可增强为：

- 每个任务强制有微任务分解
- 每个关键功能强制有 TDD 证据
- 每个 phase 结束前强制完成前验证
- 子代理的并行执行更有章法

#### 5.2.4 对门禁阶段的优化

现有 Gate 侧重结果检查。融合后可增加过程质量检查：

- 是否按规范分解任务
- 是否按要求产出工件
- 是否完成完成前验证
- 审查反馈是否闭环

#### 5.2.5 对知识沉淀阶段的优化

融合后，`@Knowledge-Curator` 不仅沉淀问题修复经验，还可沉淀：

- 常见 Harness 纪律违规模式
- 高质量执行模板
- 不同项目类型的工件裁剪策略
- TDD / 验证 / 审查的最佳实践范式

## 7. 质量保障措施

### 7.1 测试策略

融合开发应采用“四层验证”：

1. **规则层测试**：检查 Phase/Step 映射、工件模板、角色责任扩展是否正确加载。
2. **配置层测试**：检查 workflow、agent profile、chip.json 扩展字段解析是否兼容。
3. **引擎层测试**：检查 orchestrator 是否能正确注入 Harness 协议、回写证据、推进 Gate。
4. **端到端测试**：以真实项目跑完整链路，验证工件、门禁、报告和回退流程。

具体建议：

- 为 Harness Policy Engine 编写单元测试。
- 为 Artifact Registry / Evidence Collector 编写结构测试。
- 为 Gate 合规检查逻辑编写回归测试。
- 建立“最小项目样本集”用于持续集成验证。

### 7.2 性能优化方案

融合会增加文档、验证和证据收集开销，因此必须提前做好性能治理：

1. **分级执行**：按项目类型启用不同强度的 Harness 约束。
2. **模板自动生成**：由系统预填文档骨架，减少重复生成成本。
3. **证据摘要存储**：黑板中保留摘要，详细日志保存在独立目录，避免黑板膨胀。
4. **门禁按需拉取**：Gate 只拉取必要证据，避免全量扫描。
5. **异步归档**：文档归档与知识沉淀尽量放在主流程后半段异步执行。

### 7.3 风险应对机制

#### 风险 1：流程过重导致推进变慢

- **应对**：建立轻量/标准/严格三档模式。

#### 风险 2：角色执行负担增大

- **应对**：通过模板自动化、工件回填和默认策略减少手工判断。

#### 风险 3：无人值守模式下文档与验证开销过大

- **应对**：引入夜间模式阈值，对非关键工件采用延后补全机制。

#### 风险 4：旧项目兼容性不足

- **应对**：优先在新项目和 Web Feature 项目启用，旧项目采用可选接入模式。

#### 风险 5：门禁规则冲突或误报

- **应对**：设置试点期白名单和人工复核开关，逐步收敛阈值。

### 7.4 回滚策略

若融合过程中出现显著回归，建议支持以下回滚：

- Workflow 元数据回滚
- Agent Profile 责任扩展回滚
- Gate Harness 检查项按开关禁用
- Domain Chip 输出契约字段兼容回退

确保不影响现有 Loop Agent 主流程可用性。

## 8. 预期成果与评估标准

### 8.1 预期成果

融合完成后，应达到以下成果：

1. Loop Agent 获得统一的 Harness 执行纪律接入能力。
2. 各关键 phase 拥有明确的工件与证据标准。
3. 开发阶段具备更强的微任务化、TDD 化与完成前验证能力。
4. Gate 报告同时覆盖结果质量与过程合规性。
5. 黑板可追踪“做了什么、怎么做的、证据是什么、为什么通过”。
6. Knowledge Base 可沉淀融合后的最佳实践与反模式。

### 8.2 核心性能指标

建议建立以下验收指标：

#### 过程指标

- Harness 协议注入成功率 ≥ 95%
- 强制工件生成完整率 ≥ 95%
- 完成前验证证据覆盖率 ≥ 90%
- 审查反馈闭环完成率 ≥ 95%
- TDD 执行率 ≥ 80%（新功能类任务）

#### 质量指标

- Gate 1 一次通过率提升 ≥ 20%
- 回归缺陷率下降 ≥ 30%
- 需求遗漏导致的返工率下降 ≥ 20%
- 发布前高优先级问题数下降 ≥ 30%

#### 效率指标

- 在标准模式下，总交付周期增长不超过 15%
- 在严格模式下，总交付周期增长不超过 25%
- 无人值守模式夜间成功跑通率 ≥ 85%

### 8.3 输出质量标准

融合后系统输出需满足：

- 文档齐全：强制工件无缺失。
- 状态一致：黑板、报告、Artifact Registry 数据一致。
- 证据充分：关键完成声明可回溯到命令、测试、反馈或报告。
- 门禁闭环：所有 Gate 均能说明“是否通过”和“为什么通过/不通过”。
- 可维护：新增规则和模板具备清晰裁剪策略与扩展说明。

### 8.4 验收方法

建议采用“三段式验收”：

1. **文档验收**：检查规则、模板、字段、目录和执行说明是否完整。
2. **技术验收**：检查 workflow、agent、blackboard、chip、引擎改造是否生效。
3. **项目验收**：选取真实项目跑通，验证 Phase 推进、Gate 判断、工件产出、证据回写和最终交付质量。

验收结论可分为：

- `PASS`：正式纳入默认流程
- `PASS_WITH_ACTIONS`：允许上线，但需在下一迭代修正问题
- `FAIL`：回退到试点模式继续优化

## 9. 融合目标与验收清单

本章节用于把“融合方向”进一步收敛为“可执行目标 + 可验证清单 + 可否决门槛”，确保融合后的系统不只是流程上看起来完整，而是真正具备全自动闭环操作、生产级代码交付、直接部署落地、Token 高效利用和严格按预设逻辑执行的工程能力。

### 9.1 融合总目标

融合完成后的目标系统，必须同时满足以下五类总目标：

1. **全自动闭环目标**：从需求输入到代码、文档、测试、门禁、发布准备完成，形成可追踪、可回退、可恢复、可审计的自动化闭环。
2. **生产级交付目标**：输出代码必须满足生产环境质量要求，而不是停留在 demo、样例或半成品级别。
3. **直接部署目标**：最终产物必须具备部署所需的配置完整性、验证完整性和发布完整性，能够直接进入部署流程。
4. **Token 效率目标**：在保证质量的前提下，控制上下文膨胀、避免重复推理、减少无效执行和无意义循环。
5. **流程收敛目标**：系统必须严格按照预设逻辑推进，避免出现 phase 漂移、角色越权、重复执行、空转和伪完成。

### 9.2 全自动闭环目标与验收清单

#### 9.2.1 目标定义

“全自动闭环”不是简单地把多个步骤串起来，而是要求系统具备“自动启动 → 自动分派 → 自动执行 → 自动验证 → 自动回退 → 自动归档 → 自动交付”的连续能力，并保证关键状态始终可追踪。

#### 9.2.2 必须实现的闭环能力

- 自动读取或初始化黑板。
- 自动识别项目类型、任务类型、风险等级和 Phase 入口。
- 自动为任务注入 Harness 执行协议。
- 自动生成阶段工件骨架并在执行过程中增量回填。
- 自动收集验证证据、测试结果和审查反馈。
- 自动根据 Gate 结果推进、阻断或回退流程。
- 自动沉淀知识条目、文档索引和发布说明。
- 自动在异常中执行重试、降级、回滚或待人工确认分支。

#### 9.2.3 验收清单

- [ ] 用户仅提供需求与基本约束后，系统可自行推进至可交付状态。
- [ ] 任一 Phase 结束后，黑板、工件、证据和状态同步一致。
- [ ] 任一 Gate 失败后，系统能自动给出回退路径和修复入口。
- [ ] 任一 Agent 中断后，系统能基于黑板和工件恢复上下文继续执行。
- [ ] 无人值守模式下，系统可在不打断用户的前提下连续推进多个 phase。
- [ ] 最终输出中包含代码、文档、报告、证据和下一步部署输入。

### 9.3 生产级代码交付目标与验收清单

#### 9.3.1 目标定义

融合后的系统必须产出“可维护、可测试、可审查、可部署、可回归”的完整代码，而不是只满足“看起来功能可运行”。

#### 9.3.2 生产级代码标准

- 代码结构清晰，模块职责边界明确。
- 所有关键能力均有测试覆盖和验证证据。
- 配置、环境变量、依赖、脚本、构建流程齐全。
- 无高危安全问题、无明显性能瓶颈、无未处理的关键异常路径。
- 输出具备可读性、可维护性和后续演进空间。

#### 9.3.3 验收清单

- [ ] 新功能代码具备明确目录结构、模块边界和依赖关系。
- [ ] 关键路径具备单元测试、集成测试或端到端测试覆盖。
- [ ] 关键业务逻辑具备失败测试、通过测试和重构记录或等效证据。
- [ ] 无 Blocker 级安全问题、无 P0/P1 级功能缺陷。
- [ ] 构建、启动、测试、校验脚本可在项目内直接运行。
- [ ] 所有强制文档工件齐全并与代码版本一致。

### 9.4 直接部署落地目标与验收清单

#### 9.4.1 目标定义

“可直接部署”意味着交付产物已经完成部署前的必要准备，而不是要求用户再手工补环境、补脚本、补配置、补说明。

#### 9.4.2 直接部署能力要求

- 部署配置完整。
- 构建产物完整。
- 环境变量说明完整。
- 数据库、API、静态资源、依赖服务等部署前条件完整。
- 部署验证、冒烟验证、回滚说明完整。

#### 9.4.3 验收清单

- [ ] 交付物中包含完整部署说明或部署脚本入口。
- [ ] 所有运行时依赖、环境变量和第三方服务接入均已列出。
- [ ] 至少完成一次构建验证和一次部署前冒烟验证。
- [ ] 最终 Release Notes 中包含部署前提、部署步骤和回滚策略。
- [ ] 不存在“仅在本地可运行、缺少生产环境说明”的半成品状态。
- [ ] 输出结果能直接交由 `@DevOps` 进入上线流程。

### 9.5 Token 使用效率目标与验收清单

#### 9.5.1 目标定义

Token 优化不是单纯减少输出，而是在确保质量的前提下，降低重复分析、上下文膨胀和无效循环，把推理资源投入到真正影响结果的决策点。

#### 9.5.2 优化原则

- **黑板优先**：状态、结论、依赖和决策优先写入黑板，减少重复回忆。
- **工件优先**：后续阶段优先读取结构化工件，而不是反复从长文本中二次推理。
- **摘要优先**：长日志、长报告进入摘要存储，Gate 和 Orchestrator 默认读取摘要。
- **增量优先**：只处理变更部分，不重复重跑已通过的检查。
- **边界优先**：只有在关键决策点才展开深推理，普通任务执行采用模板化与参数化策略。

#### 9.5.3 验收清单

- [ ] 同一阶段不会因缺少状态记录而重复完整推理两次以上。
- [ ] Gate 检查默认基于结构化摘要，不依赖全量长文本重复扫描。
- [ ] 工件生成采用模板回填，不重复生成相同章节主体内容。
- [ ] 无人值守模式下连续多轮运行不出现明显上下文失控。
- [ ] 长链路任务中，Orchestrator 的上下文只保留调度级信息，不承载全部领域细节。
- [ ] 系统具备“无进展检测 + 自动中止/回退”机制，避免空转消耗。

### 9.6 严格按预设逻辑执行目标与验收清单

#### 9.6.1 目标定义

融合后的系统必须表现为“确定性的受控流程引擎”，而不是“凭即时判断随机发挥的代理集合”。所有关键行为都必须在规则、phase、gate、黑板和角色职责范围内运行。

#### 9.6.2 必须满足的逻辑约束

- `@Orchestrator` 只做调度，不做领域内容代写。
- Agent 只能在授权 phase、授权职责、授权工件范围内行动。
- 未通过 Gate 的任务不得进入下一阶段。
- 缺少关键工件、关键证据或关键依赖时必须阻断而不是跳过。
- 任何自动修复、自动重试、自动回退都必须留痕。

#### 9.6.3 验收清单

- [ ] 所有 phase 转移均可在黑板或状态机日志中追溯。
- [ ] 任意角色执行结果都能追溯到对应输入、职责和输出工件。
- [ ] 任何 Gate 放行都具备明确通过依据，而非软性判断。
- [ ] 任意失败恢复都具备重试次数、回退原因和当前状态记录。
- [ ] 不存在绕过 Gate、跳过工件、缺证据宣告完成的情况。
- [ ] 不存在角色越权直接替代其他角色核心职责的情况。

### 9.7 流程防偏离机制与验收清单

#### 9.7.1 偏离类型定义

需重点防范以下偏离类型：

- **流程偏离**：跳过 phase、跳过 gate、顺序错乱。
- **角色偏离**：Orchestrator 做执行、执行者做终审、审查者改业务代码。
- **目标偏离**：开始做与需求无关的扩展、优化或探索性内容。
- **资源偏离**：高 Token 消耗但无进展，高迭代次数但无有效产出。
- **结果偏离**：输出 demo 级结果却宣称达到生产级交付。

#### 9.7.2 防偏离机制

- 在 workflow 中显式声明 phase 依赖与禁止跳跃规则。
- 在 agent profile 中声明职责边界和禁止行为。
- 在 blackboard 中记录目标、约束、当前 phase、当前 gate 和 blocking reason。
- 在 Gate 中加入“目标一致性检查”“工件完整性检查”“证据充分性检查”。
- 在无进展检测中加入“重复输出”“重复失败”“重复分析”的识别策略。

#### 9.7.3 验收清单

- [ ] 连续 3 轮无有效进展时，系统能自动报警、暂停或回退。
- [ ] 当任务输出开始偏离目标时，系统能通过目标一致性检查识别异常。
- [ ] 当角色尝试越权时，系统能阻断并回归标准分工。
- [ ] 当缺少关键输入时，系统优先阻断或请求补充，而不是臆造补全。
- [ ] 当生成结果达不到生产级标准时，系统不会错误标记为完成。

### 9.8 最终上线前总验收清单

以下清单建议作为融合正式启用前的最终门槛：

- [ ] 已完成至少 1 个真实 Web Feature 项目的完整试点。
- [ ] 已跑通从需求到发布准备的全自动闭环链路。
- [ ] 已验证强制工件集、证据集、报告集的完整性。
- [ ] 已验证 Gate 失败、重试、回退、恢复场景。
- [ ] 已验证无人值守模式下的夜间连续执行稳定性。
- [ ] 已验证 Token 优化策略不会破坏结果质量与流程完整性。
- [ ] 已通过 Final-Reviewer 的融合终审。
- [ ] 已形成可供团队复用的知识条目和标准模板。

### 9.9 一票否决项

存在以下任一情况时，融合版本不得进入默认流程：

- 无法稳定生成完整工件链。
- Gate 能被绕过或存在“无证据放行”。
- 无法在失败后基于黑板恢复执行。
- 产物仍然停留在 demo 级，无法满足部署要求。
- Token 消耗明显失控且无法通过优化策略收敛。
- 无人值守模式中出现长时间空转、重复执行或伪完成。

## 11. 最终建议

综合分析，推荐采用以下落地策略：

1. **以 Loop Agent 为唯一总控入口**，避免双入口冲突。
2. **将 boss-auto-harness 定位为执行纪律和工件标准层**，而不是第二套流程引擎。
3. **优先改规则、模板、配置，再改引擎**，先低成本验证价值。
4. **先在 Web Feature 场景试点**，验证后再扩展到更多项目类型。
5. **把 Harness 证据链纳入 Gate 与黑板**，形成真正可审计的研发闭环。
6. **同步建设知识沉淀机制**，将融合过程中的经验快速标准化，反哺全局规则库。

若按本计划推进，Loop Agent 将从“强流程编排系统”升级为“强编排 + 强执行纪律 + 强证据链 + 强工件治理”的一体化自动研发平台，显著提升复杂项目中的稳定性、透明度与交付质量。
