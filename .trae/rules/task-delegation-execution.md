# 任务拆分与专职 Agent 立即执行规则

> **版本**：v1.1
> **生效日期**：2026-06-20
> **适用范围**：Loop-Harness-Agent 全部会话与项目
> **优先级**：最高（与 `loop-agent.md` 同级，任何会话不得绕过）
> **性质**：全局规则，必须永久记忆并严格执行

---

## 一、核心铁律（不可违反）

收到用户指令后，AI **必须立即** 对任务进行系统性拆分与分析，并启动专职 Agent 落地执行，**不得停留在派发阶段**。

- 拆分不是可选项，是强制第一步。
- 派发不是终点，必须立即启动执行。
- 等待用户确认才能执行，只有在触及架构变更、安全合规、资源超阈（>150% 预算）、不可逆操作时方可例外。

---

## 二、拆分标准（六维拆解）

对每个用户指令，必须在收到指令后的第一轮响应中完成以下拆解，并以清单形式呈现：

1. **目标（Goal）**：子任务要解决什么问题、交付什么结果、成功标准是什么。
2. **范围（Scope）**：明确边界，不做什么比做什么更重要，防止范围蔓延。
3. **优先级（Priority）**：P0（阻塞/立即）> P1（必须/当日）> P2（增值/择机）。
4. **依赖（Dependency）**：哪些子任务必须串行，哪些可以并行；阻塞项必须先行解决。
5. **资源（Resource）**：需要的代码路径、API、密钥、端口、Agent 类型、外部服务、域名等。
6. **验收标准（Acceptance）**：完成时必须具备什么证据（测试通过、截图、日志、报告、lint 结果等）。

---

## 三、Agent 分配原则

按任务性质精准匹配专职 Agent，严禁越权或一肩挑：

| 任务类型 | 专职 Agent | 说明 |
|----------|------------|------|
| 任务总控 / 拆分分派 | `@Orchestrator` | 只做路由，不做领域推理 |
| Bug / 线上故障 / 应急修复 | `@Bug-Defect-Repairer` | 禁止同时承担新功能开发 |
| 纯编码实现（已知方案） | `@Dedicated-Coding-Engineer` | 按 Spec/Plan 执行 |
| 代码审查 / Gate 1 | `@Code-Reviewer` | 禁止生成者自审 |
| 性能压测 / Gate 2 | `@Professional-Performance` | 输出量化指标 |
| 全栈测试 / Gate 3 | `@全栈测试员` | 输出测试报告 |
| 文档生成 / 归档 | `@Professional-Documenter` | 按模板输出 |
| 架构设计 / 技术选型 | `@Architect-Engineer` | 输出架构文档 |
| 后端服务 / API / 数据库 | `@Backend-Engineer` | 输出代码 + API 文档 |
| 前端页面 / 组件 / UI | `@Frontend-Developer` | 输出可运行页面 |
| 部署运维 / CI/CD / Nginx / SSL | `@DevOps-Deployment-Engineer` | 输出部署脚本与配置 |
| 需求分析 / PRD | `@Requirements-Analyst` | 输出 Product-Spec.md |
| 产品决策 / 版本规划 | `@Product-Manager` | 业务拍板 |
| 交互设计 / 用户旅程 | `@UX-Researcher` | 输出 UX 流程 |
| 视觉设计 / 组件库 | `@UI-Designer` | 输出设计稿 |
| 知识沉淀 / 6 段式教程 | `@Knowledge-Curator` | 输出知识条目 |
| 最终终审 / Gate 4 | `@Final-Reviewer` | 风险判定 |

## 四、立即执行 checklist（每轮必做）

收到用户指令后，按以下顺序执行，缺一不可：

- [ ] **Step 1 读取上下文**：读取 `项目进度记录.md`、相关 Spec/Plan、历史会话记忆。
- [ ] **Step 2 拆分任务**：用六维拆解法列出所有子任务，标注 P0/P1/P2。
- [ ] **Step 3 分配 Agent**：为每个子任务指定唯一专职 Agent，禁止跨领域兼职。
- [ ] **Step 4 明确输入输出**：每个子任务必须包含：输入、输出、验收标准、优先级、依赖、资源。
- [ ] **Step 5 并行启动**：无依赖的 P0/P1 任务必须在同一轮内同时启动（使用 `Task` 工具并发调用）。
- [ ] **Step 6 建立反馈机制**：明确每个 Agent 完成后返回什么、由谁汇总、超时多久未响应要升级。
- [ ] **Step 7 监控进度**：Agent 失败或阻塞时，立即重试 / 回退 / 升级 / 请求人工确认，禁止静默等待。
- [ ] **Step 8 更新黑板**：每完成一个子任务或修复一个 Bug，立即更新 `项目进度记录.md`。

---

## 五、禁止行为（红线）

违反以下任一行为，视为流程失控：

1. ❌ 只拆分不执行，或拆分后停留在派发阶段。
2. ❌ 把任务拆分后等待用户确认再启动（除非涉及架构变更、安全合规、资源超阈、不可逆操作）。
3. ❌ Agent 失败不做处理，静默等待或占位替代。
4. ❌ 一个 Agent 包揽多个专业领域任务（例如让 `@Backend-Engineer` 同时做 UI 设计）。
5. ❌ Orchestrator 做领域推理或替代执行者验证。
6. ❌ 缺少验收证据就宣告完成。
7. ❌ 连续 3 轮无进展不触发告警、不回退、不升级。
8. ❌ 不更新黑板就进入下一轮或宣告阶段完成。

---

## 六、与 Loop-Harness-Agent 协同

本规则作为 `loop-agent.md` 的强制补充，优先级同级：
- `@Orchestrator` 负责拆分与分派，只做路由，不做领域推理。
- 各专职 Agent 负责落地执行，一 Agent 一领域。
- 每完成一个子任务必须更新 `项目进度记录.md`。
- 门禁失败必须回到对应 Phase 修复，不得绕过。
- 任何子任务完成前必须具备 Fresh Evidence（当场验证的测试、日志、截图、报告等）。

## 七、违规处理

一旦检测到违反本规则：
1. 立即停止当前流程，不继续推进。
2. 输出明确的偏离告警，说明违反了哪一条。
3. 执行恢复动作：补齐拆分 → 补齐 Agent 分配 → 补齐执行 → 补齐证据 → 更新黑板。
4. 将违规事件记录进 `项目进度记录.md` 的“偏离与恢复记录”。
