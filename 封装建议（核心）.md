## Phase 0：工程底座搭建（1小时，立即能做）
先把架子搭起来，这是所有后续工作的基础，Trae Solo有标准的目录约定，严格按这个来。

### 具体操作
1. **创建标准工程目录**
在你的项目根目录下，一次性建好这套结构：
```text
your-project/
├── .trae/                    # Trae 核心配置目录
│   ├── trae.toml             # 全局主配置
│   ├── agents/               # 16角色Agent配置（后续放这里）
│   └── skills/               # Skill资产库（后续放这里）
├── blackboard/               # 全局黑板（共享状态）
│   ├── state.json            # 全局状态快照
│   ├── input/                # 输入目录（放PRD）
│   ├── output/               # 最终交付物
│   └── .snapshots/           # 状态快照备份
├── workflows/                # 编排蓝图
└── config/                   # 全局配置（预算、质量标准）
```

2. **写好基础版 trae.toml**
在 `.trae/trae.toml` 里先配置核心能力，把黑板模式、并行能力、基础护栏打开：
```toml
version = "1.0"
name = "auto-dev-pipeline"

[system]
blackboard_enabled = true     # 启用黑板（基底通信的核心）
persist_state = true          # 状态持久化，支持断点续跑
parallel_agents = 8           # 最大并行Agent数，先保守开8个
max_iterations = 100          # 全局最大迭代次数（第一道硬刹车）
max_budget_usd = 20.0         # 全局预算上限（第二道硬刹车）

[optimization]
ralph_mode = true             # 全局默认Ralph模式：每轮重置上下文
enable_caching = true         # 启用结果缓存
```

3. **验证环境**
在终端执行 `trae status`，确认配置加载成功、黑板模式正常启用。

### 本阶段验收标准
✅ 目录结构完整  
✅ Trae 启动正常，无配置报错  
✅ Blackboard 读写正常

## Phase 1：跑通最小闭环（1天，最关键的一步）
**不要一上来就做16个角色全流程**，先跑通最小的「Maker-Checker」循环，验证 Loop + Harness 的模式在Trae里跑得通。这一步跑通了，后面只是复制扩展。

### 我们选最小验证单元：后端开发 + 代码审查门禁
也就是：给需求 → 后端Agent写代码 → 自动调用代码审查Harness → 不通过就自动修复 → 循环直到通过。

### 具体操作步骤
#### Step 1：先写第一个 Harness Skill（代码审查门禁）
在 `.trae/skills/gate1-code-review/` 下创建 `SKILL.md`，这就是你的第一个可复用资产：
```markdown
# Skill: Gate1 代码质量审查门禁
## 用途
独立校验后端代码质量，作为开发循环的停止条件。
## 检查标准（必须全部满足才算通过）
1. 无Blocker级安全漏洞
2. 单元测试覆盖率 ≥ 80%
3. 代码符合ESLint/GoLint规范，0错误
4. 无硬编码密钥、无SQL注入风险
5. 圈复杂度单函数 ≤ 10
## 输出格式
必须返回固定JSON：
{
  "passed": true/false,
  "blocker_issues": [],
  "fix_suggestions": []
}
## 执行规则
- 只做审查，不修改代码
- 严格按标准判定，不放宽
- 不通过必须明确列出问题和修复方向
```

#### Step 2：写第一个 Agent Profile（后端工程师）
在 `.trae/agents/backend.agent.toml` 定义后端Agent，让它**自带内部小Loop + 绑定上面的Harness Skill**：
```toml
name = "backend-engineer"
type = "specialist"

[system]
stateless = true              # Ralph模式：每轮重置上下文
worktree = true               # 独立工作树，避免污染主代码
max_turns = 8                 # 单任务最多迭代8轮（内部循环护栏）

[skills]
# 绑定上面写的门禁Skill
bound_skills = ["gate1-code-review"]

[prompt]
system = """
# 角色：后端开发工程师
## 内部循环规则
1. 读取需求，编写代码
2. 自动调用 gate1-code-review Skill 进行审查
3. 不通过：根据修复建议修改代码，回到第2步
4. 通过：输出最终代码，结束任务
## 停止条件
gate1-code-review 返回 passed = true
"""
```

#### Step 3：手动触发，跑通第一次循环
给它一个简单需求（比如“写一个用户登录接口，包含参数校验”），验证：
- 它会不会自动调用审查Skill
- 审查不通过会不会自动修复
- 通过了会不会正常停止
- 会不会无限循环（8轮上限有没有生效）

### 本阶段验收标准
✅ 输入一个简单接口需求，Agent自动循环开发  
✅ 不通过门禁时自动修复，最多重试8轮  
✅ 通过门禁后自动停止，不会空转  
✅ 整个过程无需人工介入

> 这一步是整个体系的“元模式”——所有角色、所有阶段的本质都是「执行 → 过门禁 → 不通过就修复」的小循环。这个跑通了，剩下的都是规模化复制。

## Phase 2：全角色能力补齐（2天）
最小闭环验证通过后，把剩下的角色和门禁全部按同样的模式补齐。

### 具体操作
#### 1. 把4层质量门禁全部做成独立Skill（半天）
按Phase 1的模板，依次写完另外3个门禁Skill：
- `gate2-performance.harness.skill`：性能压测门禁（P95、错误率、吞吐量标准）
- `gate3-testing.harness.skill`：全栈测试门禁（0 P0/P1 Bug、覆盖率要求）
- `gate4-final.harness.skill`：最终终审门禁（全流程核验、风险评估）

所有Harness都做成**无状态、纯校验、输出固定格式**的Skill，任何Agent、任何Loop都能调用。

#### 2. 把16个角色全部做成Agent Profile（1.5天）
按分层架构，逐个创建Agent配置文件，每个都遵循统一模板：
- 顶层：`orchestrator.agent.toml`、`product-manager.agent.toml`
- 业务层：`requirements.agent.toml`、`ux-researcher.agent.toml`、`ui-designer.agent.toml`
- 架构层：`architect.agent.toml`
- 开发层：`backend.agent.toml`、`frontend.agent.toml`、`bug-fix.agent.toml`
- 质量层：`code-reviewer.agent.toml`、`performance-engineer.agent.toml`、`tester.agent.toml`
- 知识层：`knowledge-curator.agent.toml`
- 交付层：`documenter.agent.toml`、`final-reviewer.agent.toml`、`devops.agent.toml`

每个Agent遵守三个统一规则：
1. **无状态**：`stateless = true`，每轮重置上下文（Token优化+避免上下文污染）
2. **窄输入**：只读取Blackboard里自己需要的路径，不全量读取
3. **契约输出**：输出必须符合固定Schema，写入Blackboard指定路径

### 本阶段验收标准
✅ 4个门禁Skill全部可独立调用  
✅ 16个Agent配置完成，每个都能独立执行单任务  
✅ 每个Agent都有明确的输入输出契约

## Phase 3：全链路自动化编排（1天）
单个角色都能跑了，接下来用Orchestrator把它们串起来，实现从PRD输入到生产输出的全自动流转。

### 具体操作
#### 1. 编写Workflow蓝图
在 `workflows/prd-to-production.json` 里定义完整的阶段流转、依赖关系、门禁节点：
```json
{
  "name": "prd-to-production-v1",
  "phases": [
    {
      "id": "REQUIREMENTS",
      "tasks": [
        {"agent": "product-manager", "deps": []},
        {"agent": "requirements-analyst", "deps": ["product-manager"]}
      ],
      "exit_gate": "requirements-baseline"
    },
    {
      "id": "DESIGN",
      "tasks": [
        {"agent": "ux-researcher", "deps": ["requirements-analyst"]},
        {"agent": "ui-designer", "deps": ["ux-researcher"]}
      ]
    },
    {
      "id": "DEVELOPMENT",
      "parallel": true,
      "tasks": [
        {"agent": "backend-engineer", "deps": ["architect"]},
        {"agent": "frontend-engineer", "deps": ["architect"]}
      ],
      "exit_gate": "gate1-code-review"
    },
    // ... 后续质量、文档、终审、部署阶段
  ],
  "global_guardrails": {
    "max_budget_usd": 50,
    "max_total_turns": 200,
    "no_progress_detection": true
  }
}
```

#### 2. 配置Orchestrator总控
给Orchestrator Agent加上调度逻辑：
- 读取Workflow蓝图
- 按依赖关系自动派发任务
- 监控每个任务状态，失败自动重试
- 到达门禁节点自动调用对应Harness Skill
- 触发硬刹车条件时紧急停止

#### 3. 端到端跑一次全流程
放一份简单PRD到 `blackboard/input/`，启动全流程，观察：
- 阶段是否按顺序流转
- 并行任务是否同时执行
- 门禁是否正常拦截
- 异常时是否能重试/熔断

### 本阶段验收标准
✅ 输入PRD后，全流程自动触发，无需人工分步指令  
✅ 阶段间自动流转，门禁节点自动校验  
✅ 全局护栏生效：超预算/超迭代/无进展时自动停止  
✅ 失败任务自动重试，连续失败触发熔断

## Phase 4：资产化与持续优化（1-2天）
全流程跑通后，做价值升华，把流水线变成可复用、自进化的资产。

### 核心做三件事
#### 1. Token优化落地
- 所有Agent强制开启Ralph模式（每轮重置）
- 严格执行最小输入原则：只传文件路径，不传全量内容
- 启用结果缓存：相同输入的任务直接复用历史结果
- 监控每个阶段的Token消耗，找出优化点

#### 2. 知识沉淀系统上线
- 配置Knowledge-Curator Agent自动监听全流程
- Bug修复、架构决策、踩坑记录自动沉淀为知识库条目
- 下次遇到相似问题自动匹配推荐解决方案

#### 3. 打包为Domain Chip（可选，终极形态）
把整套Workflow + Agent + Skill + 知识库打包成一个「Web功能开发领域芯片」，对外只暴露「输入PRD → 输出生产代码」两个接口，内部完全黑盒化。后续新项目直接调用这个Chip，不用重复搭流水线。
