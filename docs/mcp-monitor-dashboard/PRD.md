## 一、文档概述

### 1.1 文档目的
本 PRD 将 Product-Spec 中定义的产品愿景转化为**可执行、可测试、可追溯**的详细需求规格，为后续架构设计、开发实现、测试验收提供唯一依据。

### 1.2 文档范围
- **包含范围**: M1 MVP 的全部 P0 功能（F001-F006）
- **排除范围**: M2/M3 功能、非功能性需求的详细实现方案

### 1.3 术语定义
| 术语 | 定义 |
|------|------|
| **Agent** | Loop-Harness-Agent 中的 16 个专职角色（PM、Requirements、Architect 等） |
| **MCP Server** | 基于 Python 的 MCP 协议服务端，提供 15 个工具接口 |
| **WebSocket (WS)** | 全双工通信协议，用于后端向前端实时推送数据 |
| **黑板** | 共享内存区域，存储项目进度、任务状态、证据等协作数据 |
| **Phase** | 开发阶段，共 10 个（INIT → DEPLOY） |
| **Gate** | 质量门禁，共 4 个（代码审查/性能/测试/终审） |

## 二、用户故事（User Stories）

### US-001: 查看所有 Agent 实时状态
**作为** 项目负责人，
**我希望** 在一个页面内看到所有 16 个 Agent 的当前运行状态，
**以便于** 快速掌握全局进度，及时发现异常 Agent。

**验收标准 (AC)**:
- [ ] 页面加载后 2 秒内显示所有 16 个 Agent 状态卡片
- [ ] 每个 Agent 卡片显示：名称、角色图标、当前状态（idle/running/error/complete）
- [ ] running 状态的 Agent 显示当前执行的任务描述和进度百分比
- [ ] error 状态的 Agent 显示错误信息摘要
- [ ] 状态每 2 秒自动刷新一次（无需手动操作）
- [ ] 支持按状态筛选（全部/idle/running/error/complete）

**优先级**: P0 (Must Have)
**复杂度**: 中等
**预估工时**: 4h

---

### US-002: 查看操作日志实时流
**作为** 技术负责人，
**我希望** 看到所有 Agent 的操作日志按时间顺序实时滚动显示，
**以便于** 追踪任务执行过程，快速定位问题根因。

**验收标准 (AC)**:
- [ ] 日志以时间轴形式展示，最新日志在底部自动滚动到可视区域
- [ ] 每条日志显示：时间戳、Agent 名称、级别（info/warn/error/debug）、消息内容
- [ ] 日志推送延迟 < 500ms（从产生到前端展示）
- [ ] 支持 3 种过滤方式：
  - 按 Agent 过滤（下拉选择单个或多个 Agent）
  - 按级别过滤（仅显示 error/warn/info/debug）
  - 按关键词搜索（支持模糊匹配）
- [ ] 错误级别日志以红色高亮显示
- [ ] 警告级别日志以黄色高亮显示
- [ ] 默认保留最近 1000 条日志，超出部分归档（可配置）
- [ ] 点击单条日志可展开查看完整 metadata

**优先级**: P0 (Must Have)
**复杂度**: 高
**预估工时**: 6h

### US-003: 查看任务执行进度
**作为** 项目负责人，
**我希望** 以可视化进度条查看当前任务的执行进度，
**以便于** 直观评估剩余工作量，预测完成时间。

**验收标准 (AC)**:
- [ ] 进度条显示百分比数字（如 "67%"）
- [ ] 进度条颜色随进度变化：
  - 0-30%: 红色（刚开始）
  - 31-70%: 黄色（进行中）
  - 71-99%: 绿色（即将完成）
  - 100%: 蓝色（已完成）
- [ ] 进度更新时使用平滑过渡动画（时长 300ms，ease-out 缓动函数）
- [ ] 显示当前任务名称和预计剩余时间（如有）
- [ ] 支持同时显示多个 Agent 的进度（并行任务场景）
- [ ] 进度数据准确性 100%（与 MCP Server 返回值一致）

**优先级**: P0 (Must Have)
**复杂度**: 低
**预估工时**: 3h

### US-004: 查看项目全景仪表盘
**作为** 项目负责人，
**我希望** 在一个面板中看到项目的关键指标和整体健康状况，
**以便于** 快速判断项目是否正常推进。

**验收标准 (AC)**:
- [ ] 展示以下核心指标：
  - 项目名称和当前 Phase（如 "Phase 5: Development"）
  - Phase 总体进度（环形进度图 + 百分比）
  - 任务统计：总数 / 已完成 / 进行中 / 失败
  - Token 预算使用情况（已用 / 总量 / 百分比 + 进度条）
  - Agent 统计：总数量 / 运行中 / 空闲 / 异常
  - 4 个 Gate 状态：pending(灰) / passed(绿) / failed(红)
- [ ] 数据准确性 100%（与黑板数据一致）
- [ ] 关键指标变化时有微动画提示（如数字滚动效果）
- [ ] 点击任意指标卡片可跳转到对应详情页

**优先级**: P0 (Must Have)
**复杂度**: 中等
**预估工时**: 5h

### US-005: 控制 MCP 系统生命周期
**作为** 运维工程师，
**我希望** 通过看板界面直接启动、停止、重启 MCP 系统，
**以便于** 无需命令行操作即可管理系统运行状态。

**验收标准 (AC)**:
- [ ] 提供三个控制按钮：
  - **启动** (绿色): 启动 MCP Server 进程
  - **停止** (红色): 停止 MCP Server 进程
  - **重启** (黄色): 先停止再启动
- [ ] 按钮状态根据当前系统状态动态启用/禁用：
  - 未运行 → 启用【启动】按钮，禁用【停止】【重启】
  - 运行中 → 禁用【启动】，启用【停止】【重启】
  - 启动/停止过程中 → 所有按钮禁用，显示 loading 动画
- [ ] 操作前弹出确认对话框（防止误操作）
- [ ] 操作结果即时反馈（成功/失败 toast 提示）
- [ ] 操作响应时间 < 1秒（UI 反馈延迟）
- [ ] 操作日志记录到操作日志流中（US-002）

**优先级**: P0 (Must Have)
**复杂度**: 中等
**预估工时**: 3h

### US-006: 建立 WebSocket 实时连接
**作为** 系统（技术需求），
**我需要** 前端通过 WebSocket 与后端建立持久连接并接收实时数据推送，
**以便于** 实现 ≤ 2秒的数据更新频率，避免轮询浪费资源。

**验收标准 (AC)**:
- [ ] 页面加载后自动建立 WebSocket 连接（无需用户手动触发）
- [ ] 连接成功后立即收到初始全量数据（Agent 列表 + 项目概览）
- [ ] 之后每 2 秒收到增量更新数据
- [ ] 连接断开时自动重连机制：
  - 首次重连：立即尝试
  - 后续重连：指数退避（1s, 2s, 4s, 8s, ... 最大 30s）
  - 最大重连次数：无限（直到手动关闭页面）
  - 重连期间显示"连接中断，正在重连..."提示
- [ ] 心跳检测：客户端每 30 秒发送 ping，服务端回复 pong
- [ ] 连续 3 次心跳超时（90s）判定连接失效，触发重连
- [ ] 支持多浏览器标签页同时打开（每个标签独立连接）

**优先级**: P0 (Must Have)
**复杂度**: 高
**预估工时**: 4h

#### 3.1.2 交互细节
- **鼠标悬停**: 显示 Agent 详细信息 tooltip（角色描述、最近操作时间、当前任务详情）
- **点击卡片**: 跳转到该 Agent 的详情页面（M2 功能，M1 仅高亮选中态）
- **状态筛选器**: 页面顶部横向 Tab 栏（全部 / 运行中 / 空闲 / 异常 / 完成）
- **排序**: 默认按角色编号排序，支持按状态分组（同状态的聚在一起）

#### 3.1.3 数据结构
```typescript
interface AgentCardData {
  id: string;                    // "product-manager"
  displayName: string;           // "Product Manager"
  roleType: string;              // "planning"
  icon: string;                  // emoji 或 SVG 图标路径
  status: AgentStatus;
  currentTask?: {
    name: string;                // "编写 Product-Spec"
    phase: number;               // 当前所处 Phase
    progress: number;            // 0-100
    startTime: Date;
    estimatedEnd?: Date;
  };
  lastActivityTime: Date;
  errorSummary?: string;         // 仅 status=error 时有值
}

type AgentStatus = 'idle' | 'running' | 'error' | 'complete';
```

#### 3.1.4 状态流转规则
```
idle ──→ running ──→ complete
  ↑        ↓
  └── error ──→ idle (修复后)
         ↓
       running (重试)
```

  - `info`: 默认文字颜色（#333）

- 日志渲染使用虚拟滚动（React Window 或类似库）
- 可视区域外日志不渲染 DOM
- 10,000 条日志首次渲染 < 200ms
- 新增单条日志插入 < 16ms（60fps）

### 3.4 F006: WebSocket 实时连接

#### 3.4.1 连接生命周期
```
页面加载
  ↓
创建 WebSocket 实例 (ws://localhost:8000/ws)
  ↓
监听 onopen 事件
  ↓
发送 { type: 'subscribe', channels: ['agent_status', 'logs', 'overview'] }
  ↓
接收初始全量数据 { type: 'init', data: {...} }
  ↓
进入消息循环:
  ├── 接收 { type: 'agent_status_update' } → 更新 Agent 卡片
  ├── 接收 { type: 'log_entry' } → 追加到日志流
  ├── 接收 { type: 'project_overview' } → 更新仪表盘
  ├── 接收 { type: 'alert' } → 触发告警
  └── 每 30s 发送 { type: 'ping' } ← 收到 { type: 'pong' }
  ↓
检测到 onclose 或 onerror
  ↓
触发重连流程（指数退避）
  ↓
重连成功 → 恢复消息循环
重连失败 → 继续重连（无限循环直到页面关闭）
```

#### 3.4.2 消息协议
```typescript
// 客户端 → 服务端
type ClientMessage =
  | { type: 'subscribe'; channels: string[] }
  | { type: 'unsubscribe'; channels: string[] }
  | { type: 'ping' }

// 服务端 → 客户端
type ServerMessage =
  | { type: 'init'; data: InitPayload }
  | { type: 'agent_status_update'; data: AgentStatus }
  | { type: 'log_entry'; data: OperationLog }
  | { type: 'project_overview'; data: ProjectOverview }
  | { type: 'alert'; data: AlertInfo }
  | { type: 'pong' }
  | { type: 'error'; message: string }
```

## 四、非功能性需求

### 4.1 性能需求
| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 首屏渲染时间 (FCP) | ≤ 2.0s | Lighthouse |
| 最大内容绘制 (LCP) | ≤ 2.5s | Lighthouse |
| 累积布局偏移 (CLS) | ≤ 0.1 | Lighthouse |
| WebSocket 消息延迟 | ≤ 500ms (P95) | 服务端打时间戳对比 |
| API 响应时间 | ≤ 200ms (P95) | 服务端日志 |
| 内存占用（前端） | ≤ 150MB | Chrome DevTools |
| 内存占用（后端） | ≤ 100MB | ps/top 命令 |

### 4.2 兼容性需求
- Chrome ≥ 120, Firefox ≥ 121, Safari ≥ 17, Edge ≥ 120
- WebSocket API 必须可用（不支持降级到 SSE 或轮询）
- ES2022+ 语法支持（Vite 编译处理）
- CSS Grid + Flexbox 布局支持

### 4.3 可用性需求
- 系统可用性 ≥ 99%（计划内停机除外）
- 单点故障恢复时间 < 5 分钟（人工介入）
- 数据持久化：SQLite 自动备份每小时一次

### 4.4 安全性需求
- 所有 API 输入必须经过验证和清洗
- WebSocket 连接需校验 Origin 头（防止 CSRF）
- 敏感信息（Token、密钥）不在前端日志中明文展示
- 生产环境强制 HTTPS/WSS

### 4.5 可维护性需求
- TypeScript 严格模式开启
- 代码注释率 ≥ 30%
- 组件/函数单一职责
- ESLint + Prettier 强制代码风格
- 单元测试覆盖率 ≥ 80%

## 五、数据需求

### 5.1 数据来源
| 数据项 | 来源 | 更新频率 | 格式 |
|--------|------|---------|------|
| Agent 状态 | MCP Server `get_status` 工具 | 2秒/次 | JSON |
| 操作日志 | 黑板 `项目进度记录.md` | 实时追加 | Markdown → 结构化 JSON |
| 项目概览 | 黑板 `state.json` | 2秒/次 | JSON |
| 进度数据 | Agent 执行引擎内部状态 | 2秒/次 | JSON |

### 5.2 数据流向
```
MCP Server (Python)
  ↓ get_status() / read_blackboard()
后端聚合层 (FastAPI)
  ↓ 定时采集 (2s interval) + 事件驱动
WebSocket 广播
  ↓ ws.send()
前端 React 组件
  ↓ setState() / useReducer()
DOM 渲染
```

### 5.3 数据存储（后端缓存）
- SQLite 数据库用于历史日志持久化
- 内存缓存用于当前状态（Redis 可选升级）
- 日志保留策略：默认 30 天，可配置
