## 一、产品定位

### 1.1 产品定义
MCP 实时监控看板系统是 Loop-Harness-Agent 的配套可视化监控平台，**实时展示所有 Agent 的运行状态、操作日志、执行进度与项目全景数据**，实现与 MCP 系统的生命周期同步管理。

### 1.2 核心价值主张
- **实时透明化**：将 16 个 Agent 的"黑盒运行"变为"白盒可视化"
- **全链路追踪**：从任务派发到交付验收的完整生命周期可视化
- **主动预警**：偏离检测、Token 超预算、门禁失败等异常实时告警
- **决策支持**：项目负责人通过数据驱动调度优化与资源分配

### 1.3 差异化定位
| 维度 | 现有方案 | 本产品优势 |
|------|---------|-----------|
| 数据获取 | 需手动调用 MCP 工具 | WebSocket 自动推送（≥ 2秒/次） |
| 可视化 | 命令行文本输出 | React 图表 + 进度条 + 实时面板 |
| 状态同步 | 异步轮询 | 事件驱动实时更新 |
| 多端访问 | 仅 IDE 内部 | 桌面/移动响应式 Web 应用 |
| 历史追溯 | 黑板文件手动查看 | 时间轴回放 + 趋势分析 |

## 五、数据模型

### 5.1 核心实体

```typescript
// Agent 状态
interface AgentStatus {
  id: string;                    // Agent ID (e.g., "product-manager")
  name: string;                  // 显示名称
  role: string;                  // 角色类型

  error?: string;                // 错误信息

// 操作日志
interface OperationLog {
  id: string;
  agentId: string;
  timestamp: Date;
  level: 'info' | 'warn' | 'error' | 'debug';
  message: string;
  metadata?: Record<string, any>;
}

// 项目全景
interface ProjectOverview {
  projectName: string;
  currentPhase: string;           // 当前 Phase
  phaseProgress: number;         // Phase 进度 (0-100)
  totalTasks: number;            // 总任务数
  completedTasks: number;        // 已完成任务数
  failedTasks: number;           // 失败任务数
  tokenBudget: {
    used: number;
    total: number;
    percentage: number;
  };
  agents: {
    total: 16;
    active: number;
    idle: number;
    error: number;
  };
  gates: {
    gate1: 'pending' | 'passed' | 'failed';
    gate2: 'pending' | 'passed' | 'failed';
    gate3: 'pending' | 'passed' | 'failed';
    gate4: 'pending' | 'passed' | 'failed';
  };
}

// WebSocket 消息类型
type WSMessage =
  | { type: 'agent_status_update'; data: AgentStatus }
  | { type: 'log_entry'; data: OperationLog }
  | { type: 'project_overview'; data: ProjectOverview }
```

## 六、用户流程

### 6.1 主流程：打开看板 → 监控 → 操作

```
1. 用户打开浏览器访问 http://localhost:3000
   ↓
2. 前端自动建立 WebSocket 连接到后端
   ↓
3. 后端返回初始状态（Agent 列表 + 项目概览）
   ↓
4. 用户看到四象限看板：
   ├── 左上：Agent 状态矩阵（16 个卡片）
   ├── 右上：项目全景仪表盘
   ├── 左下：操作日志实时流
   └── 右下：当前任务进度条
   ↓
5. WebSocket 每 2 秒推送最新数据
   ↓
6. 用户可点击任意 Agent 卡片查看详情
   ↓
7. 用户可通过控制面板启动/停止 MCP 系统
```

### 6.2 异常流程：告警触发

```
1. 后端检测到异常（如 Token 超预算）
   ↓
2. 立即通过 WebSocket 推送 alert 消息
   ↓
3. 前端收到 alert，触发：
   ├── 右上角红色告警徽章 + 数字
   ├── 对应 Agent 卡片变红闪烁
   └── 告警面板新增一条记录
   ↓
4. 用户点击告警查看详情和处理建议
```

## 十二、关键决策记录

### 决策 D001: 技术栈选择
- **决策内容**: 前端采用 React + TypeScript + Vite，后端采用 FastAPI + WebSocket
- **决策时间**: 2026-06-19
- **决策依据**:
  - 来源1: 用户明确要求"React 框架结合 TypeScript 开发"
  - 来源2: MCP 系统本身为 Python 生态，FastAPI 与之无缝集成
  - 来源3: Vite 在 2026 年已成为主流构建工具，HMR 性能优异
- **备选方案**:
  - 方案A: Vue3 + Vite → 未选原因：用户已明确 React
  - 方案B: Next.js SSR → 未选原因：纯 SPA 场景，SSR 增加复杂度
- **置信度**: 🟢 高确信
- **影响范围**: 全栈技术选型
- **可逆性**: 中等成本（需重写大部分代码）

### 决策 D002: 数据存储选择
- **决策内容**: 使用 SQLite 作为持久化存储
- **决策时间**: 2026-06-19
- **决策依据**:
  - 来源1: 单实例部署场景，无需分布式数据库
  - 来源2: SQLite 零配置、嵌入式、备份简单（复制文件即可）
  - 来源3: Python 标准库 sqlite3 无需额外安装
- **备选方案**:
  - 方案A: PostgreSQL → 未选原因：过重，需额外维护数据库服务
  - 方案B: JSON 文件 → 未选原因：并发读写性能差，无查询能力
- **置信度**: 🟢 高确信
- **影响范围**: 数据层设计
- **可逆性**: 低成本（ORM 抽象层可切换）

### 决策 D003: 实现范围确定 (MVP)
- **决策内容**: v1.0 只做 P0 功能（F001-F006），P1/P2 留给后续版本

## 十三、技术风险预判

### 风险 R001: WebSocket 连接稳定性
- **风险描述**: 网络波动导致 WebSocket 断连，数据丢失
- **发生概率**: 中
- **影响程度**: 严重（核心功能不可用）
- **缓解方案**:
  - 前端实现自动重连机制（指数退避）
  - 心跳检测（30秒一次）
  - 断线重连后增量拉取丢失数据
- **备选方案**: 降级为 SSE (Server-Sent Events) 或长轮询

### 风险 R002: MCP Server 进程管理
- **风险描述**: 外部进程启停可能失败或僵死
- **发生概率**: 中
- **影响程度**: 中等（部分功能不可用）
- **缓解方案**:
  - 进程健康检查定时器（每 10 秒）
  - 超时自动 kill + 重启
  - 启动/停止操作的幂等性保证
- **备选方案**: 使用 Supervisor/systemd 管理进程

### 风险 R003: 大量日志数据的渲染性能
- **风险描述**: 日志量过大导致前端渲染卡顿
- **发生概率**: 高（长时间运行必然出现）
- **影响程度**: 中等（用户体验下降）
- **缓解方案**:
  - 虚拟滚动（仅渲染可视区域）
  - 日志分页/窗口化（最近 1000 条）
  - 搜索/过滤在后端执行
- **备选方案**: 日志归档到独立页面，主面板只显示摘要
