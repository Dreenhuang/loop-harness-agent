## 1. 文档目标

本文档用于将融合计划书中的目标、门槛和验收要求固化为独立标准，作为后续技术改造、试点运行、终审放行和默认启用的统一依据。

该标准重点保障以下结果：

- 融合后的流程具备全自动闭环运行能力。
- 融合后的系统能输出符合生产级标准的高质量完整代码。
- 交付结果具备直接进入部署流程的条件。
- Token 消耗可控，避免上下文膨胀、重复推理和无意义空转。
- 所有关键行为严格按照预设 phase、gate、角色边界和黑板状态推进。

---

## 2. 融合总体验收目标

融合版本必须同时满足以下五个顶层目标：

1. **全自动闭环目标**：从需求输入到代码、文档、测试、门禁、发布准备完成，形成可追踪、可恢复、可回退、可审计的自动化闭环。
2. **生产级交付目标**：最终输出代码达到生产环境质量标准，不接受 demo 级、样例级或半成品级结果。
3. **直接部署目标**：交付结果具备部署所需的配置、说明、验证和回滚信息，能够直接进入 `@DevOps` 发布流程。
4. **Token 效率目标**：系统在保证质量前提下避免重复推理、长上下文堆积和无效循环。
5. **流程收敛目标**：系统严格按预设逻辑执行，不发生 phase 漂移、角色越权、缺证据放行和伪完成。

---

## 3. 全自动闭环验收标准

### 3.1 验收目标

系统必须具备从“接收需求”到“形成可交付结果”的连续闭环能力，并能在过程中自动记录、自动验证、自动阻断和自动恢复。

### 3.2 必备能力

- 自动读取或初始化黑板。
- 自动识别任务类型、风险等级和 phase 入口。
- 自动向角色派发 Harness 执行协议。
- 自动生成工件骨架并增量回填。
- 自动收集证据、测试结果和审查反馈。
- 自动推进、阻断或回退 Gate。
- 自动写回知识、索引和发布说明。

### 3.3 验收清单

- [ ] 用户仅提供需求与约束后，系统可自动推进至可交付状态。
- [ ] 任一 Phase 完成后，黑板、工件、证据和状态保持一致。
- [ ] 任一 Gate 失败后，系统可自动给出修复入口和回退路径。
- [ ] 任一 Agent 执行中断后，系统可基于黑板恢复执行。
- [ ] 无人值守模式下，可连续推进多个 phase 且不中断主流程。
- [ ] 最终输出同时包含代码、文档、测试报告、门禁报告和部署输入。

---

## 4. 生产级代码验收标准

### 4.1 验收目标

代码输出必须满足“可维护、可测试、可审查、可部署、可回归”的生产标准，而不是仅以局部功能可运行为验收条件。

### 4.2 生产级标准

- 代码结构清晰，模块边界明确。
- 关键路径具备测试与验证证据。
- 构建、配置、环境变量、依赖与脚本齐全。
- 不存在 Blocker 级安全缺陷与 P0/P1 级功能问题。
- 输出结果可被后续团队成员稳定维护与扩展。

### 4.3 验收清单

- [ ] 新功能代码具备明确目录结构、模块职责和依赖边界。
- [ ] 关键路径具备单元测试、集成测试或端到端测试覆盖。
- [ ] 关键逻辑具备失败测试、通过测试和重构证据或等效证明。
- [ ] 无 Blocker 安全问题、无 P0/P1 级缺陷。
- [ ] 构建、启动、测试、校验脚本可直接执行。
- [ ] 强制工件齐全并与代码版本一致。

---

## 5. 直接部署验收标准

### 5.1 验收目标

交付物必须具备“无需用户再补核心部署信息”的能力，可以直接进入部署实施阶段。

### 5.2 部署前置要求

- 部署配置完整。
- 构建产物完整。
- 环境变量说明完整。
- 外部依赖、数据库、API、静态资源接入说明完整。
- 部署验证、冒烟验证与回滚说明完整。

### 5.3 验收清单

- [ ] 交付物中包含部署说明或部署脚本入口。
- [ ] 运行时依赖、环境变量和第三方服务已全部列出。
- [ ] 至少完成一次构建验证和一次部署前冒烟验证。
- [ ] `Release-Notes.md` 包含部署前提、部署步骤和回滚策略。
- [ ] 不存在“只在本地可运行”的未完成交付状态。
- [ ] 输出结果可直接交由 `@DevOps` 继续上线。

---

## 6. Token 效率验收标准

### 6.1 验收目标

系统必须在保持产出质量的同时控制 Token 消耗，避免无意义的上下文扩张、重复分析和空转循环。

### 6.2 优化原则

- 黑板优先。
- 工件优先。
- 摘要优先。
- 增量优先。
- 关键决策点深推理，普通任务参数化执行。

### 6.3 验收清单

- [ ] 同一阶段不会因缺少状态记录而重复完整推理两次以上。
- [ ] Gate 默认基于结构化摘要而非全量长文本反复扫描。
- [ ] 工件采用模板回填，不重复生成相同主体内容。
- [ ] 无人值守连续运行时不出现明显上下文失控。
- [ ] `@Orchestrator` 只保留调度级上下文，不承载全部领域细节。
- [ ] 系统具备无进展检测与自动中止/回退能力，避免空转消耗。

---

## 7. 严格按预设逻辑执行验收标准

### 7.1 验收目标

融合后的系统必须表现为确定性流程引擎，所有关键执行都受到规则、phase、gate、黑板和角色职责约束。

### 7.2 逻辑约束

- `@Orchestrator` 只做调度，不做领域内容代写。
- Agent 只在授权 phase、授权职责和授权工件范围内行动。
- 未通过 Gate 的任务不得进入下一阶段。
- 缺少关键工件、关键证据或关键依赖时必须阻断。
- 自动修复、自动重试、自动回退必须留痕。

### 7.3 验收清单

- [ ] 所有 phase 转移可在黑板或状态机日志中追溯。
- [ ] 任意角色输出都可追溯到输入、职责和输出工件。
- [ ] 任意 Gate 放行都具备明确证据基础。
- [ ] 任意失败恢复都记录重试次数、回退原因和当前状态。
- [ ] 不存在绕过 Gate、跳过工件、缺证据宣告完成的情况。
- [ ] 不存在角色越权替代其他角色核心职责的情况。

---

## 8. 流程防偏离验收标准

### 8.1 偏离类型

- 流程偏离
- 角色偏离
- 目标偏离
- 资源偏离
- 结果偏离

### 8.2 防偏离机制

- 在 workflow 中显式声明 phase 依赖与禁止跳跃规则。
- 在 agent profile 中声明职责边界与禁止行为。
- 在 blackboard 中记录目标、约束、phase、gate 和 blocking reason。
- 在 Gate 中加入目标一致性、工件完整性和证据充分性检查。
- 在无进展检测中加入重复输出、重复失败和重复分析识别。

### 8.3 验收清单

- [ ] 连续 3 轮无有效进展时，系统能自动报警、暂停或回退。
- [ ] 当输出开始偏离目标时，系统能通过一致性检查识别异常。
- [ ] 当角色尝试越权时，系统能阻断并恢复标准分工。
- [ ] 当缺少关键输入时，系统优先阻断或请求补充，而不是臆造补全。
- [ ] 当结果未达到生产级标准时，系统不会错误标记为完成。

## 9. 最终上线前总验收清单

- [ ] 已完成至少 1 个真实 Web Feature 项目的完整试点。
- [ ] 已跑通从需求到发布准备的全自动闭环链路。
- [ ] 已验证强制工件集、证据集、报告集完整性。
- [ ] 已验证 Gate 失败、重试、回退、恢复场景。
- [ ] 已验证无人值守模式下的连续执行稳定性。
- [ ] 已验证 Token 优化策略不会破坏结果质量和流程完整性。
- [ ] 已通过 `@Final-Reviewer` 的融合终审。
- [ ] 已形成标准模板和可复用知识条目。

---

## 10. 一票否决项

出现以下任一情况时，融合版本不得进入默认流程：

- 无法稳定生成完整工件链。
- Gate 可被绕过或存在无证据放行。
- 无法在失败后基于黑板恢复执行。
- 产物仍停留在 demo 级，无法满足部署要求。
- Token 消耗明显失控且无法收敛。
- 无人值守模式中出现长时间空转、重复执行或伪完成。
